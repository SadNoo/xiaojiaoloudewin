# xianyuxian 建议开发路线图

## 阶段 0：需求与合规冻结

- 完成《需求确认清单》。
- 明确许可证链条和商业分发权限。
- 明确授权 API 是接入现有服务还是新建服务。
- 确定 Windows 支持范围、品牌和发布主体。
- 输出需求规格 v1.0、验收用例和威胁模型。

退出条件：许可可实施、关键决策无空项、P0 范围已冻结。

## 阶段 1：原项目基线与可测试化

- 建立干净可重复的 Windows 开发环境。
- 固定 Python、Node、Playwright/Chromium 版本。
- 验证原项目的扫码登录、消息监听、关键词回复、订单同步和卡券发货。
- 记录基线缺陷，不把上游已有缺陷混入桌面改造。
- 给发货幂等、卡券领取、授权状态机和数据迁移补测试。

退出条件：核心流程在未桌面化前已有可重复验证基线。

## 阶段 2：架构拆分与本地安全

- 拆分 FastAPI 路由和数据库访问层。
- 引入显式数据库迁移。
- 本地 API 改为回环地址、随机端口、启动令牌。
- 使用 Windows DPAPI 保护数据加密根密钥。
- 日志脱敏、诊断包和数据目录规范化。
- 统一进程、账号任务和浏览器实例生命周期。

退出条件：业务可通过本地 UI 正常运行，局域网无法访问本地 API，敏感字段抽检无明文。

## 阶段 3：在线授权服务

- 实现授权码、套餐、设备、激活、刷新、吊销和离线票据。
- 建立所有者与个人子管理员后台、额度/权限隔离、审计、频率限制和备份。
- 实现客户端版本清单、最低版本、禁止版本、强制升级和发布签名。
- 客户端实现授权状态机、到期降级和换机解绑。
- 完成服务不可用、明确吊销、时钟异常、重复激活、越权访问和凭证重放测试。

退出条件：授权服务故障不会破坏本地数据，吊销和离线策略符合需求。

## 阶段 4：Windows 桌面化

- 创建 WebView2 桌面壳、托盘、单实例和开机启动选项。
- 内嵌并管理 Python 业务内核。
- 打包 Playwright/Chromium 所需运行时。
- 实现启动进度、异常恢复、退出确认和本地通知。
- 建立 Windows 10/11 干净虚拟机安装测试。

退出条件：用户不需命令行、Python 或 Node 环境即可安装并完成核心业务。

## 阶段 5：更新、签名与发布

- 实现版本检查、签名更新、下载校验和失败回退。
- 申请并接入 Windows 代码签名。
- 生成安装包、卸载器、隐私政策、用户协议和用户手册。
- 灰度给少量真实用户，收集经过同意的脱敏问题信息。

退出条件：发布产物可追溯、可升级、可回滚，P0 验收全部通过。

## 开发顺序原则

授权服务和桌面壳不应直接嵌入现有单体文件。先固定业务基线，再建立模块边界和本地安全，之后接入授权，最后打包发布。这样可以把“上游业务故障”“授权故障”和“桌面打包故障”分开定位。
