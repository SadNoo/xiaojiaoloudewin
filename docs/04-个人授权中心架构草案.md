# xianyuxian 个人授权中心架构草案

> 状态：范围已确认，技术参数待冻结  
> 用途：部署到 Cloud 或自有 VPS，为 Windows 客户端提供在线授权和个人子管理能力。

已确认部署基线：Debian VPS、管理专用二级域名、HTTPS；每个子管理员默认最多维护 100 个有效授权码；授权支持永久、按天数到期和按自然月到期；每个授权码绑定 1 台设备、默认不限制闲鱼账号数量；离线宽限 72 小时、心跳 15 分钟。

## 1. 系统边界

授权中心只保存以下数据：

- 所有者和个人子管理员账号、密码摘要、TOTP 配置和会话。
- 授权码摘要/索引、有效期、状态、设备上限、功能权益、备注和归属。
- 激活设备的派生标识、用户自定义设备名、系统/客户端版本、首次和最近在线时间。
- 远程吊销、解绑、授权变更、版本发布和后台操作审计。
- 客户端版本清单、下载地址、SHA-256、签名、最低版本和更新策略。

授权中心不保存闲鱼 Cookie、闲鱼密码、订单、聊天内容、收货信息、卡密、回复规则或 AI API Key。

## 2. 建议部署结构

```text
Internet
   |
HTTPS 443
   |
Caddy/Nginx（TLS、限流、安全响应头）
   |
授权 API + 管理后台服务（FastAPI）
   |-------------------|
PostgreSQL           Redis（可选）
授权/设备/审计        限流/短会话/任务锁
   |
对象存储或版本下载源（安装包与签名清单）
```

建议以 Docker Compose 作为首版部署方式：

- `api`：FastAPI，提供客户端 `/v1/` 和后台 `/admin/v1/` API。
- `web`：React 管理后台静态页面，可由 Caddy/Nginx 直接提供。
- `postgres`：持久化授权、设备和审计数据。
- `redis`：首版可选；需要多实例、严格限流或任务队列时启用。
- `reverse-proxy`：自动 HTTPS、域名、请求体限制和访问日志脱敏。
- `backup`：定时加密备份 PostgreSQL，并验证恢复流程。

建议域名路由采用单一管理专用二级域名：

- `https://license.<主域名>/admin/`：所有者和子管理员页面。
- `https://license.<主域名>/admin/v1/`：管理 API。
- `https://license.<主域名>/v1/`：Windows 客户端授权 API。
- `https://license.<主域名>/downloads/`：可选的版本下载路径；使用对象存储时改为独立下载域名。

实际二级域名通过部署环境变量配置，客户端生产构建只写入授权 API 基础地址，不写入 VPS IP。

## 3. 角色与权限

### 3.1 所有者（Owner）

- 拥有全部授权码、设备、版本、子管理员和审计权限。
- 创建、停用和重置子管理员，分配其授权码额度和权限。
- 管理全局离线宽限、设备上限、心跳周期、最低版本和禁止版本。
- 发布或撤回客户端版本。
- 轮换服务密钥、强制注销会话、查看系统健康和备份状态。

### 3.2 个人子管理员（Sub-admin）

- 只能管理自己创建或所有者分配的授权码。
- 可在额度内生成、续期、停用授权码，查看其绑定设备并执行解绑/吊销。
- 不能查看其他子管理员的授权码和设备。
- 不能创建其他管理员，不能修改全局策略、版本、签名密钥和服务配置。
- 是否允许批量导出完整授权码必须单独授权，默认禁止。

### 3.3 最小权限建议

子管理员权限拆为：`license.read`、`license.create`、`license.renew`、`license.revoke`、`device.read`、`device.unbind`、`audit.read_own`。所有者可按子管理员勾选，并设置每日生成数量、总有效授权数和最长有效期。默认总有效授权数上限为 100；达到上限后不能继续生成或将新授权置为有效。

## 4. 核心数据模型

| 表 | 关键字段 |
|---|---|
| `admin_users` | 角色、用户名、密码摘要、TOTP、状态、最后登录 |
| `admin_sessions` | 会话摘要、管理员、过期时间、撤销时间 |
| `licenses` | 码索引/摘要、状态、到期、设备上限、权益、归属管理员 |
| `license_secrets` | 如需找回授权码，保存受主密钥加密的密文；默认可不保存 |
| `activations` | 授权、设备 ID、设备名、客户端版本、最后在线、吊销时间 |
| `refresh_tokens` | token 摘要、激活设备、轮换链、过期和撤销时间 |
| `release_channels` | 稳定/测试渠道及策略 |
| `releases` | 版本号、下载地址、哈希、签名、强制/最低版本状态 |
| `audit_events` | 操作者、动作、对象、前后状态摘要、IP、结果、时间 |

授权码建议使用足够随机的不可预测值，例如 128 bit 以上随机熵，按分组格式展示。数据库使用 HMAC 索引定位授权码并保存慢哈希校验，避免数据库泄露后直接得到可使用的明文授权码。

### 4.1 授权有效期规则

- `permanent`：无固定到期时间，但可被远程吊销、停用或因版本策略拒绝。
- `days`：从服务端记录的生效时刻开始增加 N×24 小时。
- `calendar_months`：使用日历月加法，例如 2026-01-31 10:00 加 1 个自然月为 2026-02-28 10:00。
- 服务端统一使用 UTC 保存时间，后台按 `Asia/Shanghai` 展示和输入；所有到期判断以服务端时间为准。
- 未激活授权默认从首次激活时开始计时；管理员也可明确设置固定生效时间和固定到期时间。
- 对尚未到期的授权续期，默认从原到期时间继续增加；已经到期的授权续期，默认从续期操作时间重新计算。

## 5. 客户端校验流程

### 5.1 首次激活

1. 客户端生成设备派生 ID 和本机密钥对或设备秘密。
2. 用户输入授权码，客户端通过 HTTPS 调用激活接口。
3. 服务端校验授权状态、有效期、设备配额和客户端版本。
4. 服务端创建设备激活记录，返回短期访问凭证、可轮换刷新凭证和签名离线票据。
5. 客户端通过 Windows DPAPI 保存必要凭证，随后才启动本地业务内核。

### 5.2 每次启动

1. 客户端优先使用刷新凭证联网验证。
2. 验证成功：更新服务端时间锚点、离线票据、版本策略和权益。
3. 服务端明确拒绝：停止自动任务，进入激活或只读界面。
4. 服务端不可达：验证本机离线票据、设备、版本和宽限截止时间。
5. 票据有效才临时启动，并持续显示离线剩余时间、后台重试联网。

### 5.3 运行心跳与远程吊销

- 客户端每 15 分钟发送一次心跳，携带激活 ID、客户端版本和随机请求 ID。
- 服务端返回授权状态、权益变化、最低版本、公告和下一次心跳间隔。
- 明确吊销时客户端停止新的自动回复/发货任务，安全结束正在提交的原子操作，然后进入只读状态。
- 普通网络失败进入离线宽限倒计时；不得把 HTTP 401/403/409 等业务拒绝当作断网。

## 6. 版本控制

- 版本采用语义化版本号，服务端分别记录 `latest_version`、`minimum_version` 和禁用版本范围。
- 更新清单包含版本、渠道、发布时间、下载地址、文件大小、SHA-256、签名、更新说明和是否强制。
- 客户端只接受由发布公钥验证通过的版本清单和安装包。
- 低于最低版本时禁止启动自动任务，但保留更新、重新激活和数据导出入口。
- 安装包建议存放在对象存储/CDN；VPS 也可直接托管，但应限制目录和 MIME 类型。

## 7. 云端安全与运维

- 所有者必须使用高强度密码和 TOTP；子管理员建议强制 TOTP。
- 密码使用 Argon2id；后台会话 Cookie 使用 `Secure`、`HttpOnly`、`SameSite`，并防护 CSRF。
- API 采用 HTTPS、请求限流、登录失败锁定、刷新凭证轮换和重放检测。
- 云端签名私钥与数据库分离保存，限制只有授权服务进程可读。
- 管理操作审计日志追加写入，普通子管理员不能删除。
- 数据库每日自动备份，至少保留 7/30 天两档，并定期做恢复演练。
- 提供 `/health/live` 和 `/health/ready`，但不得在健康接口泄露版本、密钥或数据库详情。
- 服务不可用时客户端进入既定离线宽限，不应因为 5xx 删除设备或吊销本地凭证。

## 8. 部署前仍需确认

1. Debian VPS 的 CPU、内存、地域、SSH 管理方式和 Docker 可用情况。
2. 实际二级域名、DNS 服务商及 HTTPS 证书，是否面向中国大陆网络并涉及备案。
3. 子管理员总数、最长可签发时长和具体权限；单个子管理员有效授权码额度已确定为默认 100。
4. 安装包存储位置以及稳定/测试渠道的具体下载地址。
