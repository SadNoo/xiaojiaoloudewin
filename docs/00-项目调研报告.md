## 1. 项目结论

原项目是一个本机或服务器部署的闲鱼店铺自动回复与自动发货 Web 系统，不是 Windows 桌面客户端。它由 Python 服务端统一承载业务逻辑和 API，React 页面通过本机 HTTP 接口操作服务，SQLite 保存账号、订单、卡券、规则和系统配置。

现有代码具备较完整的闲鱼业务能力，适合作为业务内核复用；但若要做成可对外分发、通过在线授权码启用的 Windows 客户端，不能只套一个桌面窗口。还需补齐桌面壳、进程生命周期、授权服务、设备绑定、安全存储、升级、安装打包、故障恢复、隐私与合规等产品化能力。

建议保留“React 前端 + Python 业务内核 + SQLite”，第一版使用 WebView2 桌面壳承载现有前端，并把 FastAPI 仅绑定在 `127.0.0.1` 的随机本地端口。授权码只用于向独立授权服务器换取短期签名凭证；闲鱼 Cookie、卡密库存和聊天数据默认保留在用户电脑，不上传授权服务器。

## 2. 原项目技术结构

| 层级 | 当前实现 | 主要文件 | 复用判断 |
|---|---|---|---|
| 启动与任务编排 | Python、Asyncio、后台 FastAPI | `Start.py`、`cookie_manager.py` | 高，可重构启动边界 |
| 闲鱼消息监听 | WebSocket、心跳、Cookie/token 刷新 | `XianyuAutoAsync.py`、`utils/ws_utils.py` | 高，但依赖非公开 Web 接口 |
| 管理 API | FastAPI，本地 HTTP，约 160 个路由 | `reply_server.py`、`api_captcha_remote.py` | 高，需模块化和收紧监听面 |
| 数据层 | SQLite，单体数据库管理类 | `db_manager.py` | 中高，需迁移机制和敏感字段加密 |
| 登录与验证 | 扫码、账号密码、Cookie、Playwright/DrissionPage | `utils/qr_login.py`、`utils/xianyu_slider_stealth.py` | 中，打包和风控适配成本较高 |
| 自动回复 | 关键词、默认回复、商品回复、AI 回复 | `ai_reply_engine.py`、`XianyuAutoAsync.py` | 高 |
| 自动发货 | 卡券、发货规则、多规格、订单状态修改 | `order_status_handler.py`、`secure_*` | 高，但需重点回归验证 |
| 管理界面 | React + TypeScript + Vite + Tailwind | `frontend/` | 高，可直接作为桌面 UI 基础 |
| 已构建页面 | FastAPI 静态资源 | `static/` | 构建产物，不作为主要源码修改入口 |
| 部署 | Docker 或直接运行 Python | `Dockerfile*`、`docker-compose*.yml` | 桌面版不直接复用 |

补充观察：

- `reply_server.py` 约有 160 个 HTTP/WebSocket 路由，职责过于集中，后续应按认证、账号、订单、卡券、规则、设置、日志拆分。
- `db_manager.py` 包含用户、会话、闲鱼账号、关键词、AI 配置、对话、卡券、订单、商品、发货规则、通知、风险日志等表。
- 前端 README 写 React 18，但当前 `frontend/package.json` 实际使用 React 19.2.3，应以锁文件和实际依赖为准。
- 仓库保留 `frontend_backup/` 和多份构建产物，开始开发前应明确唯一前端源码并清理历史路径歧义。
- 当前会话使用 HttpOnly Cookie，Cookie 的 `secure=False` 适用于本机 HTTP，但本地服务仍必须只监听回环地址并增加随机端口/进程令牌。
- 当前配置引用闲鱼、钉钉 WebSocket、阿里云百炼、Google Gemini、OpenAI 兼容 API、第三方商品详情和邮件接口等外部服务。

## 3. 原项目已有功能清单

### 3.1 店铺与账号

- 多闲鱼账号管理、启用/停用、备注和暂停时长。
- 扫码登录、密码登录、手动 Cookie 更新。
- 登录状态、Cookie 刷新、滑块或人脸验证辅助。
- 每账号独立运行监听任务。

### 3.2 回复与 AI

- 全局关键词回复、商品专属关键词回复。
- 默认回复、单会话只回复一次、图片回复。
- AI 回复开关、模型地址、模型名、API Key、自定义提示词。
- 议价折扣比例、最大金额和轮数限制。
- AI 对话历史与商品信息缓存。

### 3.3 商品、订单与发货

- 商品同步、搜索、查看和编辑。
- 订单列表、筛选、分页、详情、导入、同步、删除。
- 单订单或批量刷新状态。
- 卡券库存和卡券组管理。
- 自动发货规则、多规格配置、延时发货。
- 手动补发：仅修改状态或匹配卡券完整发货。

### 3.4 运营与管理

- 仪表盘、订单统计和趋势分析。
- 通知渠道及账号通知绑定。
- 日志、风控日志、备份导入导出。
- 管理员初始化、本地账号登录、多用户相关接口。

## 4. 关键数据与安全边界

桌面版会处理高敏感数据：闲鱼 Cookie、闲鱼账号密码（流程中临时存在）、订单与收货信息、聊天内容、卡密库存、AI API Key、授权码和设备标识。目标设计应满足：

- 闲鱼 Cookie、卡密、AI Key 在 SQLite 中加密保存，密钥由 Windows DPAPI 保护。
- 日志默认脱敏，不记录完整 Cookie、token、密码、卡密、手机号和地址。
- 本地 API 仅绑定 `127.0.0.1`，使用随机可用端口和一次性启动令牌，禁止局域网访问。
- 授权服务器不接收闲鱼 Cookie、卡密、聊天和订单明文。
- 授权码不直接作为每次请求凭证；激活后换取短期访问凭证和签名授权声明。
- 更新包必须通过 HTTPS 下载并校验数字签名和哈希。
- 提供数据导出、清空、卸载保留/删除选择和隐私说明。

## 5. 已知技术风险

| 风险 | 影响 | 初步处理 |
|---|---|---|
| 依赖闲鱼非公开 Web API、Cookie 和页面 DOM | 平台变化可能导致登录、消息、订单功能失效 | 适配层隔离、版本探测、灰度更新、故障降级 |
| Playwright/Chromium/DrissionPage 体积大 | 安装包大、首次启动慢、杀软误报概率增加 | 明确浏览器运行时策略，签名安装包，构建回归 |
| 单体文件和共享状态较多 | 改动容易产生回归 | 先拆模块边界和补关键测试，不做大爆炸式重写 |
| SQLite 中含敏感业务数据 | 本机文件泄露造成账号和卡密风险 | 字段级加密、备份加密、权限控制、日志脱敏 |
| 在线授权影响离线可用性 | 授权服务器故障会阻断客户端 | 短期凭证 + 可配置离线宽限 + 服务降级 |
| 账号自动化触发平台风控 | 账号可能被验证、限流或限制 | 明示风险、频率限制、人工接管、操作审计 |
| 上游许可文字冲突 | 可能无法合法商业分发 | 开发前完成书面许可核验 |

## 6. 许可与合规前置条件

仓库 README 展示“MIT License”，同时声明“严禁用于商业用途”，但当前仓库未包含 `LICENSE` 文件。仅凭 README 无法稳妥确认商业二次开发和分发权。

因此以下事项是正式产品化前置条件：

1. 向 `Mxucc/xianyu-super-butler` 及其所基于的上游项目确认许可证链条。
2. 若授权码涉及收费、订阅或商业客户，取得明确书面商业授权。
3. 保留必要版权声明，建立第三方依赖和许可证清单。
4. 审核闲鱼平台规则、个人信息保护、订单/聊天数据处理与自动化行为边界。
5. 产品中加入用户可读的风险提示、隐私政策、数据删除与账号安全说明。

本文只记录工程风险，不构成法律意见。

## 7. 复用与重构建议

### 直接复用

- React 页面布局和主要业务页面。
- 闲鱼消息监听、回复规则、卡券、订单和发货业务流程。
- SQLite 表所表达的业务模型。
- FastAPI 端点的请求/响应语义。

### 重构后复用

- 将 `reply_server.py` 拆为多个路由模块。
- 将 `db_manager.py` 拆为迁移、仓储和加密存储层。
- 将闲鱼 API/页面访问封装为独立 adapter，避免业务代码直接依赖 URL 和 DOM。
- 将启动、停止、崩溃恢复和任务状态统一到桌面进程管理器。
- 将前端请求层统一增加本地进程令牌、错误码和版本兼容处理。

### 不建议原样复用

- 默认对外监听地址、Docker 面向公网的部署习惯。
- 配置文件中的固定第三方接口和示例地址。
- 未脱敏的调试日志。
- 只靠客户端代码判断授权状态的方案。
